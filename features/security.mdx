## Overview

Rexec is **secure by default**, implementing multiple layers of defense to protect your terminals and data. From JWT-based authentication to container isolation, every feature is designed with security as a priority.

<Info>
Rexec follows industry best practices including CIS Docker Benchmark, OWASP Container Security, and zero-trust principles.
</Info>

## Authentication & Authorization

### JWT Authentication

All API requests require valid JWT tokens or API keys in the Authorization header:

```bash
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

**Token Properties:**
- **Algorithm**: HS256 (HMAC-SHA256)
- **Expiry**: Configurable (default 24 hours)
- **Claims**: `user_id`, `email`, `roles`, `exp`, `iat`
- **Secret**: Configured via `JWT_SECRET` environment variable

<Warning>
**Critical**: Set a strong `JWT_SECRET` in production. Random generation at startup is insecure for multi-instance deployments.
</Warning>

### API Keys

Long-lived tokens for programmatic access:

```bash
# Generate API key in dashboard
curl -X POST https://your-instance.com/api/profile/keys \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -d '{"name": "CI/CD Integration", "expires_days": 90}'

# Use API key
curl -H "Authorization: Bearer rexec_api_abc123..." \
  https://your-instance.com/api/containers
```

**Best Practices:**
- Rotate keys every 90 days
- Use separate keys per service/environment
- Store in secrets manager, not environment variables
- Revoke immediately if compromised

### Terminal MFA Lock

From `docs/SECURITY.md:120`:

Protect sensitive terminals with per-terminal MFA:

```bash
# Lock terminal with MFA
curl -X POST https://your-instance.com/api/security/terminal/{id}/mfa-lock \
  -H "Authorization: Bearer $TOKEN"

# Accessing locked terminal requires MFA code
# Dashboard prompts for 6-digit code from authenticator

# Unlock terminal
curl -X POST https://your-instance.com/api/security/terminal/{id}/mfa-unlock \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"code": "123456"}'
```

**Use Cases:**
- Production database terminals
- Terminals with access to secrets
- Compliance requirements for privileged access
- Shared accounts needing extra protection

<Info>
Terminal MFA locks are independent from account MFA. Both can be enabled simultaneously for defense in depth.
</Info>

### API Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/security/terminal/:id/mfa-status` | GET | Check if terminal is MFA-locked |
| `/api/security/terminal/:id/mfa-lock` | POST | Enable MFA protection |
| `/api/security/terminal/:id/mfa-unlock` | POST | Remove MFA lock (requires code) |
| `/api/security/terminal/:id/mfa-verify` | POST | Verify access without unlocking |

## Container Security

From `docs/SECURITY.md:1` and `internal/container/manager.go:1069`:

### Capability Restrictions

Containers run with **minimal Linux capabilities**:

```go
// From internal/container/manager.go:1159
CapDrop: []string{"ALL"},
CapAdd: []string{
    "CHOWN",            // Change file ownership
    "DAC_OVERRIDE",     // Bypass file permission checks (for sudo)
    "FOWNER",           // Bypass owner checks
    "SETGID",           // Set group ID
    "SETUID",           // Set user ID (for su/sudo)
    "KILL",             // Send signals
    "NET_BIND_SERVICE", // Bind to ports < 1024
    "SYS_PTRACE",       // For debugging/TUI apps
},
```

**Blocked capabilities** (security-sensitive):
- `SYS_ADMIN` - Mount filesystems, load kernel modules
- `SYS_PTRACE` - Trace processes (except explicitly allowed)
- `NET_ADMIN` - Modify network configuration
- `SYS_BOOT` - Reboot system
- `SYS_MODULE` - Load/unload kernel modules

### Resource Limits

From `docs/SECURITY.md:77`:

```go
Resources: container.Resources{
    Memory:     cfg.MemoryLimit,        // Hard limit
    MemorySwap: cfg.MemoryLimit,        // Disable swap
    CPUPeriod:  100000,                 // 100ms
    CPUQuota:   (cfg.CPULimit * 100000) / 1000,
    PidsLimit:  &[]int64{512}[0],       // Fork bomb protection
},
```

**Enforced Limits:**
- **Memory**: Hard limit via cgroup, OOM killer if exceeded
- **CPU**: CFS quota prevents CPU monopolization
- **PIDs**: Maximum 512 processes per container
- **Disk**: XFS quotas (if available)

### Network Isolation

From `internal/container/manager.go:660`:

```go
func (m *Manager) ensureIsolatedNetwork() error {
    _, err := m.client.NetworkCreate(ctx, "rexec-isolated", network.CreateOptions{
        Driver: "bridge",
        Options: map[string]string{
            "com.docker.network.bridge.enable_icc": "false",  // Disable ICC
        },
    })
    return err
}
```

**Network Security:**
- Containers run in `rexec-isolated` network
- Inter-Container Communication (ICC) disabled
- Containers cannot reach each other
- Outbound internet access enabled
- No inbound access from host network

## Audit Logging

Rexec logs security-relevant events for compliance and incident response:

### Logged Events

**Authentication:**
- `user_login` - Successful login
- `user_login_failed` - Failed login attempt
- `user_logout` - User logged out
- `mfa_enabled` - MFA enabled on account
- `mfa_disabled` - MFA disabled

**Terminal Access:**
- `terminal_connected` - User connected to terminal
- `terminal_disconnected` - Connection closed
- `terminal_mfa_locked` - Terminal protected with MFA
- `terminal_mfa_unlocked` - MFA protection removed
- `terminal_mfa_access_verified` - User verified MFA to access

**Container Lifecycle:**
- `container_created` - Container created
- `container_started` - Container started
- `container_stopped` - Container stopped
- `container_destroyed` - Container permanently deleted

**Collaboration:**
- `collab_session_created` - Collaboration session started
- `collab_participant_joined` - User joined session
- `collab_participant_left` - User left session
- `collab_session_ended` - Session closed

**Recordings:**
- `recording_started` - Recording began
- `recording_stopped` - Recording ended

### Log Format

```json
{
  "timestamp": "2024-03-15T14:30:00Z",
  "event": "terminal_connected",
  "user_id": "user-123",
  "user_email": "alice@example.com",
  "container_id": "abc123...",
  "ip_address": "192.168.1.100",
  "user_agent": "Mozilla/5.0 ..."
}
```

### Querying Logs

```bash
# Get audit logs for user
curl -X GET "https://your-instance.com/api/audit?user_id=user-123&limit=100" \
  -H "Authorization: Bearer $TOKEN"

# Filter by event type
curl -X GET "https://your-instance.com/api/audit?event=terminal_connected" \
  -H "Authorization: Bearer $TOKEN"

# Date range query
curl -X GET "https://your-instance.com/api/audit?from=2024-03-01&to=2024-03-15" \
  -H "Authorization: Bearer $TOKEN"
```

<Info>
Audit logs are stored in PostgreSQL and retained according to your configured retention policy (default: 90 days).
</Info>

## Advanced Security Options

### gVisor Runtime

For maximum isolation, use gVisor (runsc):

```bash
# Install gVisor
curl -fsSL https://gvisor.dev/archive.key | sudo apt-key add -
sudo apt-get update && sudo apt-get install -y runsc

# Configure Rexec
OCI_RUNTIME=runsc

# Create container
# Uses gVisor for kernel-level isolation
```

<Warning>
gVisor does not support overlay2 disk quotas. Trade-off between security and resource management.
</Warning>

### Custom Seccomp Profile

For stricter syscall filtering, create a custom seccomp profile:

```json
{
  "defaultAction": "SCMP_ACT_ERRNO",
  "syscalls": [
    {
      "names": ["read", "write", "open", "close", "stat", ...],
      "action": "SCMP_ACT_ALLOW"
    }
  ]
}
```

Configure via Docker:

```bash
--security-opt seccomp=/path/to/profile.json
```

### AppArmor/SELinux

Mandatory Access Control for additional layer:

```bash
# AppArmor profile
--security-opt apparmor=rexec-container

# SELinux context
--security-opt label=type:rexec_container_t
```

## Troubleshooting

### MFA Issues

**Problem**: TOTP codes not working

**Solutions**:
1. Check device time sync (TOTP requires accurate time)
2. Verify correct secret in authenticator app
3. Use backup code if available
4. Contact admin to reset MFA

### Terminal Access Denied

**Problem**: "Forbidden" error when connecting to terminal

**Checklist**:
1. Verify JWT token not expired
2. Check container ownership
3. Verify collaboration session if shared
4. Check audit logs for denied access events

### Container Security Warnings

**Problem**: Docker warnings about security settings

**Resolution**:
```bash
# Check Docker version
docker --version
# Update if < 20.10

# Verify seccomp support
docker info | grep seccomp

# Check AppArmor
sudo aa-status
```
