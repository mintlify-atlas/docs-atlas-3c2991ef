## Overview

Rexec Collaboration enables multiple users to access the same terminal session simultaneously. Share your terminal with teammates for pair programming, debugging production issues, or teaching - all through secure, shareable links.

<Info>
Collaboration sessions support both **view-only** and **full control** modes, giving you flexibility in how you share access.
</Info>

## How It Works

When you start a collaboration session:

1. **Owner** creates a session from their terminal
2. **Rexec generates** a unique share code (e.g., `rexec-xyz123`)
3. **Participants** join via shareable link: `/join/rexec-xyz123`
4. **All users** see the same terminal output in real-time
5. **WebSocket broadcast** distributes output to all participants

## Starting a Collaboration Session

### From the Dashboard

1. Open any running terminal (cloud container or agent)
2. Click the **Share** button in the terminal header
3. Configure session settings:
   - **Mode**: View-only or Full control
   - **Max users**: Limit concurrent participants (default: 5)
   - **Duration**: Session lifetime in minutes (default: 60)
4. Click **Create Share Link**
5. Copy and share the generated link with participants

### Via API

```bash
curl -X POST https://your-instance.com/api/collab/sessions \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "container_id": "abc123...",
    "mode": "view",
    "max_users": 10,
    "duration_minutes": 120
  }'
```


## Session Modes

### View-Only Mode

Participants can:
- ✅ See all terminal output in real-time
- ✅ Scroll through history
- ✅ Copy text from terminal
- ❌ Send input or run commands

**Use cases:**
- Demonstrations and teaching
- Debugging sessions (watchers)
- Security-sensitive environments
- Screen sharing alternative

### Full Control Mode

Participants can:
- ✅ See all terminal output
- ✅ Send input and run commands
- ✅ Resize terminal window
- ✅ Full interactive access

**Use cases:**
- Pair programming
- Collaborative debugging
- Live coding sessions
- Interactive training

<Warning>
In full control mode, **any participant can execute commands**. Only share with trusted users.
</Warning>

## Architecture

From `internal/api/handlers/collab.go:74`:

### Shared Terminal Session

Collaboration uses a shared session model where:

```go
type SharedTerminalSession struct {
    ContainerID  string
    ExecID       string
    OwnerID      string
    Connections  map[string]*websocket.Conn  // userID -> connection
    InputChan    chan []byte  // Shared input channel
    OutputChan   chan []byte  // Broadcast output channel
}
```

**Key behaviors:**
1. **Single PTY**: All participants share one pseudo-terminal
2. **Broadcast output**: Output written once, distributed to all connections
3. **Input merging**: In control mode, input from any participant is sent to PTY
4. **Resize sync**: Terminal resizes affect all participants

### Message Broadcasting

From `internal/api/handlers/collab.go:193`:

```go
func (s *CollabSession) broadcastLoop() {
    for msg := range s.broadcast {
        s.mu.RLock()
        for _, participant := range s.Participants {
            if participant.Conn != nil {
                participant.Conn.WriteJSON(msg)
            }
        }
        s.mu.RUnlock()
    }
}
```

Messages are queued and broadcast asynchronously to prevent slow clients from blocking others.

```go
    ExpiresAt:    time.Now().Add(time.Duration(req.Duration) * time.Minute),

}
```

### Automatic Cleanup

From `internal/api/handlers/collab.go:74`:

```go
func (h *CollabHandler) cleanupLoop() {
    ticker := time.NewTicker(5 * time.Minute)
    for range ticker.C {
        h.mu.Lock()
        for code, session := range h.sessions {
            if time.Now().After(session.ExpiresAt) {
                close(session.broadcast)
                delete(h.sessions, code)
                h.store.DeleteCollabSession(ctx, session.ID)
            }
        }
        h.mu.Unlock()
    }
}
```

Expired sessions are cleaned up every 5 minutes.

```go
func (h *TerminalHandler) HasCollabAccess(ctx context.Context, userID, containerID string) bool {
    // Check in-memory sessions first
    h.collabHandler.mu.RLock()
    for _, session := range h.collabHandler.sessions {
        if session.ContainerID == containerID {
            if _, hasAccess := session.Participants[userID]; hasAccess {
                return true
            }
        }
    }
    h.collabHandler.mu.RUnlock()
    
    // Fallback to database
    session, _ := h.collabHandler.store.GetCollabSessionByContainerID(ctx, containerID)
    // Check if user is participant
}
```

Access control is enforced at WebSocket connection time.

## Troubleshooting

### Participant Can't Join

1. **Session expired**: Check expiration time in dashboard
2. **Max users reached**: Owner can increase limit or kick inactive users
3. **Container stopped**: Ensure terminal is still running
4. **Invalid share code**: Verify code wasn't mistyped

### Output Not Syncing

1. **WebSocket disconnected**: Check browser console for errors
2. **Network issues**: Test connectivity to Rexec server
3. **Session closed**: Owner may have ended session

### Performance Issues

With many participants:

1. **Reduce output**: Avoid commands with heavy output (`cat` large files)
2. **Limit participants**: Set reasonable max_users
3. **Use compression**: WebSocket compression enabled by default
