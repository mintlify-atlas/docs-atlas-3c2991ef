## Overview

Rexec Cloud Terminals provide on-demand, disposable Linux environments that spin up in seconds. Each terminal runs in an isolated Docker container with configurable resources, giving you a full Linux system accessible from your browser.

<Note>
Cloud terminals are perfect for development, testing, learning, or running quick tasks without setting up local environments.
</Note>

## How It Works

When you create a cloud terminal, Rexec:

1. **Validates** your image selection and configuration
2. **Pulls** the Docker image if not already cached (with real-time progress)
3. **Creates** a container with security hardening and resource limits
4. **Configures** persistent storage for your home directory
5. **Establishes** a WebSocket connection for real-time terminal I/O

## Creating a Cloud Terminal

### From the Dashboard

1. Navigate to **Dashboard** and click **New Terminal**
2. Select your preferred distribution from 70+ options
3. Configure resources (optional):
   - Memory: 512MB - 16GB
   - CPU: 0.5 - 8 cores
   - Disk: Up to 20GB (with XFS quota support)
4. Choose a role template (optional): Standard, Node.js, Python, Go, Rust
5. Click **Create Terminal**

The terminal opens automatically once provisioning completes.

### Via API

```bash
curl -X POST https://your-instance.com/api/containers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "dev-environment",
    "image": "ubuntu",
    "role": "node",
    "memory_mb": 1024,
    "cpu_cores": 1
  }'
```

## Container Architecture

### Security Hardening

Every container runs with comprehensive security measures (from `internal/container/manager.go:1100`):

<Tabs>
  <Tab title="Capabilities">
    ```text
    CapDrop: ALL
    CapAdd:
      - CHOWN            # Change file ownership
      - DAC_OVERRIDE     # Bypass file permissions (for sudo)
      - FOWNER           # Bypass owner checks
      - SETGID/SETUID    # For su/sudo
      - KILL             # Send signals
      - NET_BIND_SERVICE # Bind to ports < 1024
      - SYS_PTRACE       # For debugging tools
    ```
  </Tab>
  <Tab title="Filesystem">
    - **Masked paths**: `/proc/kcore`, `/sys/firmware`, `/sys/kernel`
    - **Read-only paths**: `/proc/sys`, `/proc/sysrq-trigger`
    - **Tmpfs mounts**: `/tmp`, `/var/tmp`, `/run` (with size limits)
    - **Persistent volume**: `/home/user` (survives restarts)
  </Tab>
  <Tab title="Network">
    - Isolated Docker network: `rexec-isolated`
    - Inter-container communication (ICC) disabled
    - No inbound access from other containers
    - Outbound internet access enabled
  </Tab>
  <Tab title="Resources">
    - Memory limits enforced (swap disabled)
    - CPU quota via CFS bandwidth control
    - PID limit: 512 processes (fork bomb protection)
    - Disk quotas on XFS with overlay2 driver
  </Tab>
</Tabs>

### Runtime Options

Rexec supports multiple OCI runtimes via the `OCI_RUNTIME` environment variable:

| Runtime | Description | Use Case |
|---------|-------------|----------|
| `runc` (default) | Standard Docker runtime | General purpose, full features |
| `runsc` (gVisor) | Kernel-level isolation | Maximum security |
| `kata` | Lightweight VMs | Hardware isolation |
| `runsc-kvm` | gVisor with KVM | Secure + performant |

<Warning>
gVisor (`runsc`) does not support overlay2 disk quotas. Use `runc` if disk limits are required.
</Warning>

## Session Persistence with tmux

All cloud terminals use tmux for session persistence (from `internal/container/manager.go:1243`):

```bash
# Your work persists even if you disconnect
# Sessions automatically resume when reconnecting

# Configuration in /home/user/.tmux.conf
set -g history-limit 50000
set -g mouse on
set -g default-terminal "xterm-256color"
```

**Benefits:**
- Long-running processes continue after disconnect
- Command history preserved
- Split panes and windows maintained
- Mouse support for scrolling

<Info>
The terminal handler automatically creates a `main` tmux session on first connect. It resumes this session on subsequent connections.
</Info>

## Resource Management

### Memory Limits

Memory is strictly enforced using Docker's memory cgroup:

```go
// From internal/container/manager.go:1104
Memory:     cfg.MemoryLimit,
MemorySwap: cfg.MemoryLimit,  // Disable swap
```

When a container exceeds its memory limit, the kernel OOM killer terminates processes.

### CPU Quotas

CPU limiting uses CFS (Completely Fair Scheduler) bandwidth control:

```go
// From internal/container/manager.go:1073
cpuPeriod := int64(100000)  // 100ms
cpuQuota := (cfg.CPULimit * cpuPeriod) / 1000
// CPULimit is in millicores: 500 = 0.5 CPU, 2000 = 2 CPUs
```

### Disk Quotas

Disk quotas require XFS filesystem with `pquota` mount option:

```bash
# Check if quotas are available
ls -la /var/lib/docker
# Look for "xfs" filesystem type

# Quotas are set via Docker storage-opt
storage-opt: {"size": "10G"}
```

<Note>
Disk quota support is auto-detected at runtime. If unavailable, containers get unlimited disk space from the host.
</Note>

## Terminal Features

### WebSocket Streaming

Real-time terminal I/O uses optimized WebSocket communication:

```typescript
// From internal/api/handlers/terminal.go:25
ReadBufferSize:  32KB  // Large pastes and code completion
WriteBufferSize: 32KB  // Burst output handling
EnableCompression: true
```

### Message Protocol

```json
{
  "type": "input",    // input, output, resize, ping, pong
  "data": "ls -la\n",
  "cols": 120,
  "rows": 30
}
```

### Performance Optimizations

From `internal/api/handlers/terminal.go:45`:

- **Output coalescing**: 2ms buffering reduces packet overhead
- **64KB PTY buffer**: Fewer syscalls for large output
- **Compression**: Per-message deflate for large transfers
- **Write deadlines**: 5s timeout prevents blocking on slow clients

## Lifecycle Management

### Container States

- **Creating**: Image pull and container creation in progress
- **Running**: Active and ready for connections
- **Stopped**: Exited but can be restarted
- **Destroyed**: Permanently deleted (data lost)

### Automatic Cleanup

Idle containers are cleaned up based on your plan:

- **Free tier**: 24 hours of inactivity
- **Pro tier**: 7 days of inactivity
- **Enterprise**: Configurable retention

Data in `/home/user` persists across restarts but is lost on destroy.

## Troubleshooting

### Container Won't Start

```bash
# Check image availability
curl -X GET https://your-instance.com/api/images/ubuntu/status

# Verify resource limits aren't exceeded
# (Check dashboard quotas)
```

### Slow Performance

1. Check CPU/memory limits in container settings
2. Verify disk I/O isn't saturated
3. Consider upgrading to a higher resource tier

### Connection Drops

WebSocket connections auto-reconnect with exponential backoff. Sessions persist via tmux.
